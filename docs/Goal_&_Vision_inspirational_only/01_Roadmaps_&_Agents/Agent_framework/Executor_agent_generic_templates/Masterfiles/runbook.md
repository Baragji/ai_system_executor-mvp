Executor AI — Runbook
1. Overview
This document provides the standard operating procedures for using the Executor AI framework. The AI is designed for autonomous operation, driven by configuration files and activated by a simple "Proceed" command. Its purpose is to execute development tasks in a structured, verifiable, and auditable manner.

2. Core Workflow
The operational cycle is designed to be simple, repeatable, and robust.

Define the Work:

All tasks for the AI are defined as "slices" within objectives in the config/session_plan.yaml file.

Before starting, ensure this file accurately reflects the work you want the AI to perform.

Prepare the Session:

Run the session preparation script from your terminal. This script reads the tracker, identifies the next pending item from the plan, and creates a report stub in the sessions/ directory.

Command:

npm run prepare-session

(This command is defined in tools/package.json)

Activate the AI:

After the session is prepared, activate the AI with the trigger command:

Command:

Proceed

Monitor and Review:

The AI will execute the single work item. Upon completion, it will:

Commit the code changes.

Generate a detailed report in sessions/<session_id>.md.

Update the tracker/session_tracker.yaml with the outcome.

Review the session report for the Evidence Block, which contains proof of the changes made.

3. Key Artifacts & File Locations
Plan (Input): config/session_plan.yaml

This is where you tell the AI what to do.

Policy (Input): config/platform_policy.md

This file tells the AI how to behave for your specific project type (e.g., web service vs. desktop app).

Session Reports (Output): sessions/

This is the human-readable proof of work.

Tracker (State): tracker/session_tracker.yaml

This is the AI's memory and audit log.

Telemetry Logs (Output): logs/otel-spans.ndjson

Machine-readable data for performance analysis and debugging.

Metrics (Output): artifacts/metrics.json

High-level KPIs on the AI's performance, generated by tools/metrics.js.

4. Handling Blockers (STOP Protocol)
If the AI encounters ambiguity, a failing quality gate, or an unsafe request, it will enter a BLOCKED state.

It will not commit any code.

It will generate a session report detailing the precise reason for the block and provide 2-3 actionable options for you to resolve it.

To resume work, you must resolve the blocking issue and then issue the "Proceed" command again for the same work item.

5. First-Time Use (BOOTSTRAP Protocol)
If you deploy this framework in a new project that is missing the config/ or tracker/ directories, the AI will automatically create minimal, functional versions of the essential files on its first run.

It will document this bootstrapping action in its first session report. This ensures the AI can start working immediately in any environment.
