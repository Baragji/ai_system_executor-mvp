## Architecture Decisions

### Why Preflight in `installDeps.ts` (Not Earlier)

**Decision**: Validate in `ensureDependencies()` right after parsing `package.json`

**Rationale**:
1. **Single Responsibility**: Dependency installer validates before installing
2. **Reusability**: Any caller of `ensureDependencies()` gets validation
3. **Fail-Fast**: Abort before spawning npm process
4. **Clean Error Propagation**: Bubbles up through existing error handling

**Alternative Rejected**: Validation in `runInSandbox()` before calling `ensureDependencies()` would duplicate package.json parsing and break SRP.

### Why Not Auto-Fix Versions

**Decision**: Fail with structured error, don't auto-suggest or auto-fix

**Rationale** (per consultant):
> "Why not auto-fix versions right now? Fold 'suggest a safe version' into Phase 5 as an **interrupt that requires approval**—not a silent fallback."

**Phase 5 Plan**: `INVALID_DEPENDENCY` interrupt will:
1. Pause execution at checkpoint
2. Show user: "supertest@^6.4.2 invalid. Suggest: supertest@^7.0.0?"
3. Wait for user approval
4. Resume with approved version

This keeps user in control of dependency upgrades (security/compatibility).

### Registry API vs npm CLI

**Decision**: Use npm registry API (`https://registry.npmjs.org`) directly

**Rationale**:
1. **No npm CLI dependency**: Pure Node.js `https` module
2. **Faster**: Direct API call vs spawning npm process
3. **Programmable**: JSON response, easy to parse
4. **Standard Pattern**: Used by tools like Renovate, Dependabot

**Alternative Rejected**: `npm view <pkg> versions --json` requires spawning npm process and parsing stdout.

## Phase 5 Integration Path

### How Preflight Becomes INVALID_DEPENDENCY Interrupt

**Current Behavior** (Phase A-FIX-2):
```
validateDependencies() → throws DependencyPreflightError → task aborts
```

**Phase 5 Behavior** (with orchestration):
```
validateDependencies() → throws DependencyPreflightError
  → orchestrator catches
  → writes checkpoint
  → emits INVALID_DEPENDENCY interrupt
  → pauses execution
  → UI shows: "supertest@^6.4.2 invalid. Available: 0.0.1...6.3.4, 7.0.0+. Suggest: ^7.0.0?"
  → user approves/rejects/edits
  → POST /api/sessions/:id/resume with approved version
  → orchestrator replays from checkpoint with corrected dependency
  → continues execution
```

**No Code Changes Needed**: Preflight already throws structured errors. Phase 5 orchestrator will catch and convert to interrupts.

## Known Limitations

### 1. Registry API Reliability
**Issue**: npm registry may be slow/unreachable  
**Mitigation**: 10s timeout, fail-safe (treat as validation failure)  
**Phase 5**: Retry with exponential backoff, or surface as interrupt

### 2. Private Packages
**Issue**: Private npm packages (scoped `@org/pkg`) not accessible without auth  
**Current Behavior**: Fails validation (treats as NOT_FOUND)  
**Future**: Add `.npmrc` auth support or skip validation for private scopes

### 3. Monorepo Workspaces
**Issue**: Workspace dependencies (e.g., `workspace:*`) not validated  
**Current Behavior**: `validRange()` rejects workspace protocol  
**Future**: Detect and skip workspace deps

### 4. Timeout Math Not Runtime-Validated
**Issue**: Hierarchical timeouts (180s < 900s < 3600s) not tested in real 15-minute run  
**Rationale**: Math is correct by design; preflight reduces need for long repair cycles  
**Plan**: Will validate naturally during Phase 5 execution

### What Worked Well
1. **Discovery-First**: Discovery note (WA10) identified exact integration point before coding

3. **Incremental Validation**: Ran lint/typecheck/test after each win
